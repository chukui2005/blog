# 【学习打卡】Day 7 — RabbitMQ 与 Kafka 核心知识点

> 作者：初魁 | 学习周期：Java后端面试冲刺第7天 | 适合人群：Java后端求职者

---

## 一、RabbitMQ 如何保证消息不丢失

### 问题根源

消息在到达消费者之前，可能在以下环节丢失：

1. **生产者消息未到达交换机**
2. **消息未到达队列**
3. **队列中消息丢失**
4. **消费者处理消息时丢失**

### 解决方案（三个层面）

**a. 开启生产者确认机制（Publisher Confirm）**
- 确保生产者的消息能到达队列
- 失败处理：回调即时重发、记录日志、保存到数据库定时重发

**b. 开启持久化功能**
- 交换机持久化
- 队列持久化
- 消息持久化（SpringAMQP 消息默认持久，可通过 `MessageProperties` 指定 `DeliveryMode`）

**c. 开启消费者确认机制**
- **手动确认（manual）**：业务代码结束后调用 API 发送 ack
- **自动确认（auto）**：Spring 监测 listener 代码异常，无异常返回 ack，有异常返回 nack
- **关闭确认（none）**：MQ 假定消费者成功处理，投递后立即删

## 二、消息的重复消费问题如何解决

**本质**：消费者消费消息后，发送 ack 时网络抖动，导致 MQ 以为消费失败，重新投递。

**解决方案**：**保证消费逻辑的幂等性**。

常用方法：
- **数据库唯一索引**：消费记录表加唯一约束，重复插入报错
- **Redis 防重**：消费前查 Redis 是否处理过
- **分布式锁**：并发消费时加锁保证只处理一次

## 三、死信交换机（RabbitMQ 延迟队列）

### 什么是死信

消息变成死信（Dead Letter）的场景：
- 消息被消费者**拒绝**（reject/nack），且不重新入队
- 消息**超过 TTL** 过期
- **队列满了**，新消息无法入队

### 死信交换机（DLX）

配置死信交换机后，死信会被路由到指定交换机，继续处理。

**延迟队列应用**：利用 TTL + 死信交换机实现延迟消息，如订单超时未支付自动取消。

## 四、Kafka 如何保证消息不丢失

1. **生产者**：配置 `acks=all`（所有副本都收到才算成功）
2. **Broker**：副本因子 ≥ 3 + `min.insync.replicas ≥ 2`
3. **消费者**：关闭自动提交偏移量，手动提交 offset

---

## 五、Kafka 如何保证消费的顺序性

**问题**：一个 partition 内消息有序，但多 partition 并行消费时整体无序。

**解决方案**：
- **消息发到同一 partition**：指定 key（如用户 ID），相同 key 的消息到同一 partition，有序
- **单 partition 单消费者**：避免多 consumer 并行消费打乱顺序

---

## 六、Kafka 高可用机制

**ISR（In-Sync Replicas）**：与 leader 保持同步的 follower 副本集合。

- leader 宕机：从 ISR 中选举新 leader
- follower 落后太多：被踢出 ISR
- `replication.factor ≥ 3` + `min.insync.replicas = 2`：保证至少 2 个副本同步

---

## 七、Kafka 数据清理机制

- **时间维度**：`log.retention.hours`（默认 7 天）
- **大小维度**：`log.retention.bytes`
- **删除策略**：`delete`（直接删除）或 `compact`（合并相同 key 的消息，保留最新）

---

## 八、Kafka 实现高性能的设计

1. **顺序写磁盘**：利用追加写，避免随机磁盘 IO
2. **零拷贝**：`sendfile` 系统调用，绕过用户态直接传输
3. **页缓存**：利用操作系统页缓存，不自己管理内存
4. **批量发送**：减少网络请求次数
5. **分区并行**：多 partition 并行处理
