## 一、MySQL 优化

### 1. 定位慢查询

**慢查询场景**：聚合查询、多表查询、表数据量过大、深度分页查询

**定位工具**：
- 运维工具：Prometheus、Skywalking
- MySQL 慢日志：配置 `/etc/my.cnf`，开启 `slow_query_log`，设置 `long_query_time`（默认 10 秒）

### 2. SQL 执行计划分析（EXPLAIN）

| 字段 | 说明 |
|------|------|
| type | 连接类型，性能从好到差：`system > const > eq_ref > ref > range > index > all` |
| possible_keys | 可能使用的索引 |
| key | 实际使用的索引 |
| key_len | 索引占用字节数 |
| Extra | 额外信息：`Using where; Using Index` = 覆盖索引，无需回表；`Using index condition` = 索引下推，需回表 |

**优化建议**：
- 检查 key 和 key_len 确认索引命中情况
- 通过 type 避免全表扫描（all）或全索引扫描（index）
- 根据 Extra 判断是否回表，尝试添加覆盖索引或调整返回字段

## 二、存储引擎

| 特性 | InnoDB | MyISAM | Memory |
|------|--------|--------|--------|
| 事务 | 支持 | 不支持 | 不支持 |
| 锁机制 | 行锁/表锁 | 表锁 | 表锁 |
| 外键 | 支持 | 不支持 | 不支持 |
| 文件 | `.ibd`（表结构+数据+索引）| `.frm`（表结构）`.MYD`（数据）`.MYI`（索引）| 内存存储 |

**MySQL 5.5 后默认 InnoDB**，特点：支持事务（ACID）、行级锁、外键，表数据按主键顺序存储（聚簇索引）。

## 三、索引

### 1. 索引底层数据结构

**B+ 树**（InnoDB 默认）：

- 优点：阶数更多，路径更短；非叶子节点只存指针，叶子节点存数据，磁盘读写代价低；叶子节点双向链表，便于范围查询和全表扫描

**B 树 vs B+ 树**：B 树非叶子节点和叶子节点都存数据；B+ 树数据只存于叶子节点，查询效率更稳定，范围查询效率更高。

### 2. 索引分类

- **聚簇索引（聚集索引）**：数据与索引存储在一起，叶子节点保存整行数据。必须有且只有一个（主键或第一个唯一索引）
- **二级索引（非聚簇索引）**：数据与索引分开存储，叶子节点关联主键。可以有多个

**回表查询**：通过二级索引找到主键，再通过主键到聚簇索引中查找整行数据。

### 3. 覆盖索引

**定义**：查询使用了索引，且需要返回的列在索引中全部能够找到。

**优点**：避免回表查询，提升性能。

**应用**：解决超大分页问题（`limit 9000000, 10`）。优化方案：覆盖索引 + 子查询（先查 id，再根据 id 列表查询数据）。

### 4. 索引创建原则

- 数据量较大（超 10 万）且查询频繁的表
- 常作为查询条件（where）、排序（order by）、分组（group by）的字段
- 区分度高的列优先（尽量建唯一索引）
- 字符串字段较长时可建前缀索引
- 优先使用联合索引，注意最左匹配原则
- 控制索引数量（影响增删改性能）
- 索引列不能存储 NULL 值

### 5. 索引失效场景

- 违反最左匹配原则（联合索引）
- 范围查询右边的列（如 `where a=1 and b>2 and c=3`，c 索引失效）
- 索引列进行运算或函数操作（如 `substring(name,3,2)='科技'`）
- 字符串不加单引号（类型转换）
- 模糊查询 `%` 在前（如 `like '%abc'`）
- 使用 `or` 连接条件（除非所有列都有索引）
- 数据分布导致优化器放弃索引（如全表扫描更快）

## 四、事务

### 1. 事务特性（ACID）

- **原子性**：事务不可分割，要么全部成功，要么全部失败
- **一致性**：事务前后数据保持一致
- **隔离性**：事务不受其他并发事务影响
- **持久性**：事务提交后数据永久改变

### 2. 并发事务问题与隔离级别

| 问题 | 描述 |
|------|------|
| 脏读 | 读到其他事务未提交的数据 |
| 不可重复读 | 同一事务内多次读取同一数据，结果不同 |
| 幻读 | 查询无数据，插入时发现数据已存在 |

**MySQL 默认隔离级别**：`Repeatable Read`（可重复读）

---

## 五、SQL 优化经验

1. **表设计优化**：数值类型选择合适类型（tinyint/int/bigint）；char 定长效率高，varchar 可变长效率稍低
2. **SQL 语句优化**：避免 `select *`；避免索引失效写法；用 `union all` 代替 `union`；避免 where 子句中对字段进行表达式操作
3. **架构优化**：主从复制读写分离；分库分表解决存储和访问压力
