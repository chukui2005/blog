【学习打卡】Day 6 — Redis 核心知识点

作者：初魁 | 学习周期：Java后端面试冲刺第6天 | 适合人群：Java后端求职者

一、Redis 使用场景

缓存：缓存穿透、击穿、雪崩、双写一致、持久化、数据过期与淘汰策略

分布式锁：setnx、Redisson 实现

其他：计数器、保存 Token、消息队列、延迟队列、集群

二、缓存三大问题及解决方案

1. 缓存穿透

问题：查询不存在的数据，请求直达数据库。

解决方案：

  • 缓存空值：简单但消耗内存，可能不一致
  • 布隆过滤器：内存占用少，存在误判（一般 < 5%）

2. 缓存击穿

问题：热点 key 过期瞬间，大量并发请求压垮数据库。

解决方案：

  • 互斥锁：强一致，性能较差（如 Redis 的 setnx）
  • 逻辑过期：高可用，性能优（不设 TTL，额外字段记录过期时间）

3. 缓存雪崩

问题：大量 key 同时过期，或 Redis 宕机。

解决方案：

  • 给不同 key 的 TTL 添加随机值
  • Redis 集群提高可用性（哨兵、集群模式）
  • 降级限流策略（Nginx、Gateway）
  • 多级缓存（Guava、Caffeine）

三、双写一致性

强一致性方案：Redisson 读写锁（读共享锁，写排他锁）

最终一致性方案：

  • 延迟双删：先删缓存 → 改数据库 → 延时删缓存（延时时间难确定）
  • 异步通知：MQ 或 Canal 监听 binlog 同步

四、持久化策略

1. RDB（快照）

  • 原理：fork 子进程写快照，采用 copy-on-write
  • 触发方式：save（阻塞）、bgsave（后台）、配置阈值（如 save 900 1）
  • 特点：二进制文件，体积小，恢复快，可能丢数据

2. AOF（命令日志）

  • 原理：记录每个写命令
  • 刷盘策略：
    • always：同步刷盘，可靠性高，性能差
    • everysec：每秒刷盘（默认），性能适中，最多丢 1 秒数据
    • no：操作系统控制，性能好，可靠性差
  • 重写机制：bgrewriteaof 压缩命令

五、数据过期与淘汰策略

1. 过期策略

  • 惰性删除：访问时检查并删除，对 CPU 友好，内存不友好
  • 定期删除：定时随机检查删除（SLOW 模式 10hz/25ms；FAST 模式间隔 ≥ 2ms/耗时 ≤ 1ms）
  • 实际：惰性删除 + 定期删除配合

2. 淘汰策略（内存满时）

策略 范围 算法
noeviction — 不淘汰，写操作报错（默认）
volatile-ttl 设 TTL 的 key 剩余 TTL 越小越先淘汰
allkeys-random 全体 key 随机淘汰
volatile-random 设 TTL 的 key 随机淘汰
allkeys-lru 全体 key LRU（最近最少使用）
volatile-lru 设 TTL 的 key LRU
allkeys-lfu 全体 key LFU（最不经常使用）
volatile-lfu 设 TTL 的 key LFU

使用建议：优先 allkeys-lru；冷热不明显用 allkeys-random；有置顶需求用 volatile-lru；短时高频访问用 LFU 策略。

六、分布式锁

1. 基础实现

命令：setnx（SET if not exists）

问题：锁有效期控制困难

2. Redisson 实现

  • 看门狗机制：自动续期（默认每 30 秒检查，续到 30 秒）
  • 可重入锁：Hash 结构记录线程 ID 和重入次数
  • 主从一致性问题：主节点宕机可能导致锁失效（RedLock 红锁方案：在多数实例上加锁）

七、集群方案

1. 主从复制架构

一主多从，读写分离。

数据同步：

  • 全量同步：第一次连接时，主节点 bgsave 生成 RDB 发送给从节点
  • 增量同步：从节点重启或后期同步，基于 repl_backlog 日志

2. 哨兵模式

功能：监控、自动故障恢复、通知

选主规则：断开时间 → slave-priority → offset → run id

脑裂问题：

  • 原因：网络分区导致多个 master
  • 解决：配置 min-replicas-to-write（最少从节点数）和 min-replicas-max-lag（最大同步延迟）

3. 分片集群

解决：海量数据存储、高并发写

  • 哈希槽：16384 个槽，key 通过 CRC16 取模分配
  • 数据读写：客户端访问任意节点，节点转发到正确槽位

八、Redis 为什么这么快？

  1. 纯内存操作
  2. 单线程：避免上下文切换和竞争条件
  3. I/O 多路复用：epoll 模式监听多个 Socket，高效处理网络请求
  4. Redis 6.0 优化：命令回复和请求解析使用多线程，命令执行仍为单线程

说明：本笔记基于《02-Redis篇.pptx》内容整理，涵盖 Redis 核心使用场景、缓存问题解决方案、持久化、集群及高性能原理等关键知识点。
