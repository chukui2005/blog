一、JVM内存模型（JVM Memory Model）

1.1 整体结构

JVM运行时数据区分为五个区域：

1.2 各区域详解

堆（Heap）— 线程共享

  • 存储内容：所有对象实例和数组
  • GC主战场：新生代 + 老年代
  • 新生代：Eden区 + Survivor区（S0 + S1）
    • 默认比例：Eden : S0 : S1 = 8 : 1 : 1
  • 老年代：大对象直接进入，或年龄达到15后进入

方法区（Metaspace）— 线程共享

  • JDK 1.8前：永久代（PermGen），在JVM内存
  • JDK 1.8后：元空间，使用本地内存，不易OOM

虚拟机栈（VM Stack）— 线程私有

  • 栈帧结构：局部变量表、操作数栈、动态链接、返回地址
  • StackOverflowError：递归过深
  • OOM：线程过多耗尽内存

程序计数器（PC）— 线程私有

  • 当前线程执行的字节码行号
  • 唯一不会发生OutOfMemoryError的区域

二、MySQL索引原理

2.1 为什么需要索引？

没有索引时全表扫描O(n)，建立索引后查询借助B+树达到O(log n)。

2.2 B+树 vs B树

特征 B树 B+树
叶子节点 存完整数据 只存索引，叶子链表相连
范围查询 不擅长 叶子链表，范围查询极快
查询稳定性 不稳定 稳定到叶子节点
空间利用率 低 高

2.3 聚簇索引 vs 非聚簇索引

聚簇索引：叶子节点存完整行数据，一个表只有一个（主键索引）
非聚簇索引：叶子节点存主键ID，需回表查完整数据

2.4 最左前缀原则

复合索引 (name, age, position)：

  • ✅ WHERE name = ‘xxx’
  • ✅ WHERE name AND age = ‘xxx’
  • ❌ WHERE age = 20（跳过最左列）

2.5 覆盖索引

查询的所有列都在索引中，无需回表，性能最优。

2.6 索引失效的情况

失效情况 示例
使用函数/运算 WHERE LEFT(name, 3) = ‘abc’
类型转换 字段VARCHAR，查询用INT
最左前缀中断 跳过name直接查age
LIKE以%开头 WHERE name LIKE ‘%三’
OR前后不全是索引列 WHERE name = ‘a’ OR age = 20

三、面试高频提问

Q1：对象在JVM中如何分配？

优先在Eden区分配。大对象直接进入老年代，年龄达15后进入老年代。

Q2：Full GC什么时候触发？

① 老年代空间不足；② System.gc()（不推荐）；③ 元空间不足；④ CMS并发模式失败。

Q3：MySQL为何选择B+树？

矮胖（IO次数少）、范围查询快（叶子链表相连）、查询稳定（每次到叶子）。

Q4：聚簇索引和非聚簇索引的区别？

聚簇索引叶子存完整行数据；非聚簇存主键ID需回表。
