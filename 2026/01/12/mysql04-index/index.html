索引与SQL优化

核心知识点速览

知识点 要点
索引是什么 排好序的快速查找数据结构
B+Tree MySQL默认索引结构，非叶子节点存指针，叶子节点存数据
聚簇索引 数据和索引在一起，InnoDB主键索引就是聚簇的
联合索引 最左前缀法则，索引失效原因
EXPLAIN 分析SQL执行计划，关注type/key/rows/Extra

一、索引基础

1.1 什么是索引

索引就像书的目录：没有索引要逐页翻（全表扫描），有了索引直接翻到对应页。

1.2 索引的优缺点

优点 缺点
查询速度大幅提升 占用磁盘空间
减少排序和分组的时间 增删改变慢（要同步更新索引）
唯一索引保证数据唯一性 不是所有列都要建索引

二、B+Tree数据结构

2.1 为什么不用二叉树

2.2 B+Tree的特点

B+Tree三大特点：

  1. 叶子节点存数据，非叶子节点只存索引值+指针
  2. 叶子节点形成有序链表，支持范围查询（BETWEEN、>、<）
  3. 树的高度低（一般3-4层），一次查询3-4次IO，无论多少数据

三、聚簇索引 vs 非聚簇索引

3.1 聚簇索引

数据跟着索引走，找到了索引就找到了数据。

  • InnoDB中，主键索引就是聚簇索引
  • 叶子节点直接存的是整行数据

3.2 非聚簇索引（二级索引/辅助索引）

索引里不存数据，存主键值，查到主键后再去聚簇索引查数据（回表）。

3.3 覆盖索引

如果索引里已经包含了你要查的所有字段，就不用回表了。

面试重点：尽可能使用覆盖索引，避免回表。

四、索引分类

分类 关键字 说明
主键索引 PRIMARY KEY 聚簇索引，一张表只有一个
唯一索引 UNIQUE 字段值不能重复
普通索引 INDEX 最基本的索引
联合索引 多个字段 多个字段联合组成的索引
全文索引 FULLTEXT 用于大文本搜索

五、索引失效的10种场景（面试高频）

  1. 模糊查询以%开头

  2. 联合索引违反最左前缀法则

  3. 对索引列使用函数

  4. 索引列参与了计算

  5. 类型隐式转换

  6. OR条件中有非索引列

  7. != 或 <>

  8. IS NOT NULL

  9. NOT IN

  10. 全表扫描比走索引更快（数据量太小或查了大部分数据）

六、EXPLAIN分析执行计划

返回结果重点关注：

字段 说明 好/坏
type 访问类型 const > ref > range > index > ALL
key 实际使用的索引 有值就好，NULL表示没用到
rows 扫描的行数 越少越好
Extra 额外信息 Using index（覆盖索引）最好
Using filesort（文件排序）要优化

type从好到坏：

  • const：主键或唯一索引查找，最快
  • ref：普通索引等值查找
  • range：范围查询
  • index：扫描了整个索引树
  • ALL：全表扫描，最慢
