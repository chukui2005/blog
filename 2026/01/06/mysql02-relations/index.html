约束进阶与表关系

核心知识点速览

知识点 要点
外键约束 REFERENCES，保证数据完整性
级联操作 CASCADE（级联）、SET NULL（置空）
表关系 一对一、一对多、多对多
中间表 多对多必须用第三张表关联
三大范式 1NF原子性、2NF无部分依赖、3NF无传递依赖

一、外键约束进阶

上篇简单提了外键，这篇深入讲清楚。

1.1 外键的作用

外键用来保证数据完整性：子表的外键值必须在父表中存在，避免”引用不存在的数据”。

1.2 添加/删除外键

1.3 级联操作

当父表数据被删除/更新时，子表怎么办？通过级联操作指定：

级联选项 含义
CASCADE 父表删除/更新，子表也跟着删除/更新
SET NULL 父表删除/更新，子表外键置为NULL
RESTRICT（默认） 有子表记录时，禁止删除父表记录
NO ACTION 同RESTRICT

实际开发建议：大多数情况用RESTRICT（默认），防止误删。CASCADE要慎用，不小心把关联数据全删了。

二、表与表之间的关系

数据库中，表之间有三种基本关系。

2.1 一对多（1:N）最常见的

一张表中的一条记录，对应另一张表中的多条记录。

例子：一个部门有多个员工

实现方式：在”多”的表中创建外键，指向”一”的表的主键。

2.2 多对多（M:N）

一张表中的一条记录，对应另一张表中的多条记录，反过来也一样。

例子：一个学生可以选多门课，一门课也可以被多个学生选

实现方式：需要第三张中间表，中间表存两个外键。

2.3 一对一（1:1）

一张表中的一条记录，对应另一张表中的一条记录。

例子：一个用户对应一个身份证号

关键：外键加UNIQUE约束，就是一对一。

三、三大范式

范式就是设计表时需要遵循的规范，目的是减少数据冗余。

第一范式（1NF）：字段不可再分

要求：每个字段都是不可分割的原子值。

第二范式（2NF）：在1NF基础上，非主键字段必须完全依赖于主键

要求：消除部分依赖。联合主键的情况下，每个非主键字段必须依赖于全部主键，不能只依赖一部分。

第三范式（3NF）：在2NF基础上，非主键字段不能传递依赖

要求：消除传递依赖。非主键字段必须直接依赖于主键，不能通过其他非主键字段间接依赖。

面试速记

实际开发忠告：范式是为了减少冗余，但过度的范式化会让查询变慢。实际开发中经常会有意反范式（冗余一些字段来避免多表联查），这叫”空间换时间”。面试可以说这个，加分。

四、练习题

设计一个电商系统的表结构（试试自己写）：

  1. category 商品分类表（id, name）
  2. product 商品表（id, name, price, category_id）
  3. user 用户表（id, username, phone）
  4. order 订单表（id, order_no, user_id, total_price, create_time）
  5. order_item 订单明细表（order_id, product_id, quantity, price）

提示：

  • 分类和商品是什么关系？
  • 用户和订单是什么关系？
  • 订单和商品是什么关系？
  • 合理设置外键和级联策略
